Why I Built a 2D Physics Engine in C++

March 2025 · build · 8 min read

Most of what we do in software is built on top of layers we never open. You use a game engine or a physics library and you trust that when two boxes collide, the math is right. I wanted to stop trusting and start knowing.

So I built a small 2D physics engine in C++ using SFML for window and input. The goal was not to ship a product but to implement rigid body dynamics, collision detection with AABB and circles, and constraint resolution (distance constraints, pin joints) from first principles.

// Example: applying impulse at contact
void resolveCollision(Body& a, Body& b, Vec2 normal, float depth) {
  float e = 0.5f; // restitution
  Vec2 rv = b.velocity - a.velocity;
  float velAlongNormal = rv.dot(normal);
  if (velAlongNormal > 0) return;
  float j = -(1 + e) * velAlongNormal / (a.invMass + b.invMass);
  Vec2 impulse = j * normal;
  a.velocity -= a.invMass * impulse;
  b.velocity += b.invMass * impulse;
}

The real lesson was how many design choices hide inside "simple" physics: how you integrate, how you handle stacking, how you order constraint resolution. Reading papers and code from Box2D and Chipmunk filled in the gaps that tutorials skip.

If you're the kind of person who learns by building, I recommend picking one subsystem — collision, constraints, or integration — and implementing it yourself. You don't need to ship it. Understanding is the point.
